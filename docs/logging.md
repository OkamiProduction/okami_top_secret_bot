# 📊 Логирование

Бот использует структурированное логирование на базе стандартного пакета `log/slog` (появился в Go 1.21). Логи пишутся одновременно в два потока:

- **Консоль** (stdout) — текстовый формат, удобен для просмотра через `journalctl`.
- **Файл** — JSON-формат с автоматической ротацией, подходит для машинной обработки и долгосрочного хранения.

---

## ⚙️ Конфигурация логирования

Настройки задаются через переменные окружения в файле `.env` (локально) или через секреты GitHub Actions (на сервере).

| Переменная | Значения | По умолчанию | Описание |
|------------|----------|--------------|----------|
| `LOG_LEVEL` | `debug`, `info`, `warn`, `error` | `info` | Минимальный уровень записываемых сообщений. |
| `LOG_FILE` | путь к файлу | `./bot.log` (локально), `/var/log/tgbot/bot.log` (сервер) | Путь к файлу для JSON-логов. |

Пример `.env` для локальной разработки:

```env
LOG_LEVEL=debug
LOG_FILE=./bot.log
```

На сервере эти значения передаются через workflow `deploy.yml` (генерируется `.env` из секретов).

---

## 🗂️ Ротация логов

Для предотвращения неограниченного роста файла используется библиотека [lumberjack](https://github.com/natefinch/lumberjack). Параметры ротации **зашиты в коде** (`internal/logger/logger.go`):

| Параметр | Значение | Пояснение |
|----------|----------|-----------|
| `MaxSize` | 10 МБ | Максимальный размер одного файла. При превышении создаётся новый. |
| `MaxBackups` | 5 | Хранится не более 5 старых файлов. Более старые удаляются. |
| `MaxAge` | 30 дней | Файлы старше 30 дней удаляются, даже если их меньше `MaxBackups`. |
| `Compress` | `true` | Старые файлы сжимаются gzip (`.log.gz`). |

Таким образом, общий объём логов на диске не превышает примерно `10 МБ × 5 = 50 МБ` (плюс текущий файл).

### Пример имён ротированных файлов

```text
/var/log/tgbot/
├── bot.log                      # текущий файл
├── bot-2026-04-17T10-30-00.log.gz
├── bot-2026-04-16T15-22-10.log.gz
└── ...
```

## 🔍 Просмотр логов

### Локально (во время разработки)

Логи выводятся в консоль в текстовом формате. Дополнительно они пишутся в файл, указанный в `LOG_FILE`.

```bash
# Запуск бота
go run main.go

# В другом терминале — следить за файлом
tail -f bot.log
```

### На сервере (staging/production)

Systemd перенаправляет stdout в journald, поэтому текстовые логи доступны через `journalctl`. JSON-логи лежат в `/var/log/tgbot/bot.log`.

```bash
# Текстовые логи (journald)
journalctl -u tgbot -f

# JSON-логи (файл)
tail -f /var/log/tgbot/bot.log
```

Для красивого форматирования JSON можно использовать `jq`:

```bash
tail -f /var/log/tgbot/bot.log | jq
```

## 🧹 Очистка логов

В папке `scripts/` находится скрипт `clean_logs.sh`, который полностью удаляет **все** файловые и journald-логи бота **без остановки сервиса**.

### Содержимое скрипта

```bash
#!/bin/bash
# Очистка всех логов Telegram-бота без остановки сервиса

LOG_DIR="/var/log/tgbot"
SERVICE_NAME="tgbot"

echo "🧹 Очистка файловых логов в $LOG_DIR"
rm -rf "$LOG_DIR"/*
mkdir -p "$LOG_DIR"

echo "🧹 Очистка journald-логов для сервиса $SERVICE_NAME"
# Удаляем архивные journal-файлы этого юнита
journalctl --vacuum-time=1s --unit="$SERVICE_NAME" 2>/dev/null || true

echo "✅ Логи очищены. Сам сервис не трогали."
```

### Запуск на сервере

1. Скопируйте скрипт на сервер (если его там нет):
   ```bash
   scp scripts/clean_logs.sh root@<IP>:/opt/
   ```
2. Выполните с правами `root`:
   ```bash
   chmod +x /opt/clean_logs.sh
   /opt/clean_logs.sh
   ```

После выполнения папка `/var/log/tgbot` станет пустой, текущий буфер journald тоже будет очищен (в пределах возможного). Новый лог-файл создастся автоматически при следующей записи.

## 📝 Структура JSON-лога

Каждая запись в файле — одна строка JSON. Пример:

```json
{"time":"2026-04-17T10:15:30.123Z","level":"INFO","msg":"Получено сообщение","user":"okami","chat_id":123456789,"text":"/start"}
```

**Поля:**
- `time` — временная метка в UTC (RFC3339).
- `level` — уровень (`DEBUG`, `INFO`, `WARN`, `ERROR`).
- `msg` — текстовое сообщение.
- Остальные поля — дополнительные атрибуты, добавленные через `slog.String`, `slog.Int64` и т.д.

---

## 🔧 Изменение параметров ротации

Если потребуется изменить размер файла, количество бэкапов или срок хранения, отредактируйте структуру `lumberjack.Logger` в файле `internal/logger/logger.go`:

```go
fileWriter := &lumberjack.Logger{
    Filename:   logFile,
    MaxSize:    10,   // мегабайты
    MaxBackups: 5,
    MaxAge:     30,   // дни
    Compress:   true,
}
```

После изменения нужно пересобрать бинарник и задеплоить его.
